1. 개요

우리카드 회원이 우리카드 APP 내 프로모션 페이지를 통해 LFmall로 유입되었을 때, 우리카드사가 발급한 난수코드(우리카드 회원식별 코드)로 회원을 식별하고, LFmall에서의 5가지 행동(유입 / 회원가입 / 쿠폰 다운로드 / PUSH 동의 / 구매)에 따라 우리카드 포인트를 적립·정산하는 제휴 마케팅 프로젝트입니다.

핵심 과제는 "LFmall이 식별할 수 없는 우리카드 회원의 난수코드"를 LFmall의 회원·비회원 행동과 매핑해, 익월 정산 시점에 정확한 포인트 지급 대상자 리스트를 우리카드사에 전달하는 데이터 플로우를 안정적으로 설계하는 것입니다.

📌 본 설계의 전제 조건 ① 우리카드 회원에게 발급되는 난수코드는 회원당 고유한 unique 값이며, 변경되지 않는 영구 식별 코드입니다.
② 동일 회원은 언제 LFmall에 진입하더라도 항상 같은 난수코드로 유입됩니다. (단, 우리카드APP 내 프로모션 페이지를 통해 유입시만)
③ 따라서 난수코드와 LFmall 회원ID는 1:1 매핑 구조로 설계됩니다.

2. 목적

비즈니스 목적

  • 우리카드 회원 대상 LFmall 신규 유입 확대
  • 신규 회원가입 및 PUSH 마케팅 동의 확보
  • 쿠폰·구매 활동을 통한 거래액 증대

기술적 목적

  • 난수코드 - LFmall 회원ID 매핑 데이터 구축
  • 5가지 행동 이벤트별 적립 규칙 자동화
  • 월 정산 데이터 자동 추출 파이프라인 구축

아래 순서도는 ① 우리카드 APP에서 LFmall 진입 → ② 난수코드 수집 → ③ 회원/비회원 분기 → ④ 행동별 이벤트 처리 → ⑤ 재유입/재행동 처리 → ⑥ 월 정산까지의 전체 흐름을 표현합니다.

⚠️ 재유입 처리 핵심 규칙 ① 난수코드는 회원당 고유한 영구 식별 코드이므로 매핑 테이블은 난수코드 : 회원ID = 1:1 구조입니다.

4. 난수코드 URL 전달 방식 - 쉽게 설명하기

4-1. 한 줄 요약

💡 핵심 개념 우리카드사는 LFmall 주소 뒤에 "?물음표 + 키=값" 형태로 난수코드를 끼워 넣어 보내고, LFmall은 그 부분을 읽어서 DB에 저장합니다. 이걸 기술 용어로 "URL 쿼리 스트링(Query String) / URL 파라미터"이라고 부릅니다.

4-2. 실제 URL이 어떻게 생겼는지

https://www.lfmall.co.kr/event/wooricard?wc_code=aB3xK9mP2qR7&utm_source=wooricard_app&ts=1736832000&sig=8f3a9e...
구성 요소의미예시
도메인 + 경로LFmall 어디로 이동할지www.lfmall.co.kr/event/wooricard
? (물음표)"여기부터 데이터 시작" 표시?
wc_code=aB3xK9mP2qR7핵심: 우리카드 회원식별 난수코드키=값
& (앰퍼샌드)"또 다른 데이터 추가" 표시&
ts=1736832000발급 시각 (위변조 방지용)UNIX timestamp
sig=8f3a9e...위조 방지 서명 (HMAC)해시값

4-3. 우리카드사가 LFmall에 난수코드를 전달하는 절차

  1. 회원 인증 : 우리카드 회원이 우리카드 APP에 이미 로그인되어 있는 상태
  2. 난수코드 조회 : 우리카드 서버에서 해당 회원의 고유 난수코드를 조회 (회원당 1개, 영구 보관)
  3. URL 조립 : LFmall 주소 뒤에 ?wc_code=난수코드값 을 붙임
  4. 버튼에 연결 : "LFmall 방문하기" 버튼을 누르면 위 URL로 브라우저/웹뷰가 열림
  5. LFmall 수신 : LFmall 서버는 들어온 주소에서 wc_code 값을 읽어 자기네 DB에 저장
  6. 정산 시 역추적 : 익월에 LFmall이 "난수코드 ABC123이 이런 행동을 했어요"를 보내면, 우리카드사는 자체 DB에서 회원을 찾아 포인트 지급

4-4. 기술적 근거 (표준)

  • RFC 3986 (URI 표준) : URL의 "?" 뒤 query 영역은 리소스에 추가 정보를 전달하는 표준 방식으로 정의되어 있습니다. (전 세계 모든 웹의 기본 규약)
  • HTTP GET 메서드 : 브라우저가 페이지를 열 때, 서버는 URL에 포함된 query string을 자동으로 수신합니다. 별도 통신이 필요 없는 가장 가볍고 검증된 방식입니다.
  • UTM 파라미터와 동일한 원리 : 광고에서 흔히 쓰는 utm_source=facebook, utm_campaign=spring과 정확히 같은 메커니즘이며, 키 이름과 값을 비즈니스 목적(회원 식별)으로 정의해 쓰는 것뿐입니다.
  • OAuth 2.0 Authorization Code Flow : 카카오/네이버 로그인에서 콜백 URL에 ?code=xxx로 인증 코드를 전달하는 방식과 동일한 원리입니다.
⚠️ 보안상 반드시 지켜야 할 사항 ① 통신은 HTTPS 필수 (URL 내용을 중간에서 훔쳐볼 수 없게 암호화)
② 난수코드 자체는 개인정보가 유추되지 않는 무작위 문자열 (UUID, 32바이트 이상 권장)
③ 영구 난수코드 특성상 노출 위험에 대비해 HMAC 서명(sig)과 발급시각(ts)을 같이 보내고, LFmall은 만료 시간(예: 발급 후 24시간) 초과 시 거부
④ 난수코드는 회원당 고정값이므로 유출 시 영구 위험이 발생하므로, 양사 모두 저장 시 해시 또는 암호화 적용
⑤ LFmall은 난수코드를 DB에 저장할 때 평문이 아닌 해시 처리된 값으로 저장 권장

5-3. 필수 체크포인트

  • 난수코드-회원 매핑 시점: 비회원 상태에서 수집된 난수코드가 이후 회원가입/로그인 시 정확히 회원ID와 연결되는가? (쿠키 만료, 다른 디바이스 사용 시 처리)
  • 크로스 디바이스 한계 명시: 모바일 우리카드 APP → PC LFmall 진입 등 디바이스 전환 시 쿠키 단절 발생 → 우리카드 APP에서는 가급적 LFmall APP/모바일웹으로 직진 유도
  • 난수코드 위변조 방지: 서명(sig) 검증 미통과 시 저장 거부 및 로그 적재
  • 중복 적립 방지: 정산 배치 SQL에 DISTINCT/GROUP BY 정확 적용, 단위 테스트 필수
  • 주문 취소 처리: 적립 대상 송신 후 취소 발생 시 다음 달 차감 또는 사전 차감 룰 협의
  • 개인정보 보호: 우리카드사에 회원ID/이름/연락처 전달 금지, 난수코드 + 행동데이터만 전송
  • 정산 데이터 포맷 합의: 우리카드사와 컬럼 명세서, 송신 주기, 인코딩, 암호화 방식 사전 합의
  • 실패 시 재처리: 송신 실패 케이스 대비 재전송 큐, 알람 모니터링 구축
  • 법무 검토: 제휴 마케팅 동의 문구, 개인정보 제3자 제공 동의서 사전 검토
  • QA 시나리오: 신규/기가입/재유입/취소/디바이스 전환 등 최소 20개 시나리오 테스트

5-4. 우리카드사와 사전 협의 항목

6. 기대효과

📈 정량 효과

  • 외부 회원 식별 표준 프레임워크 구축

7-2. 대표 시나리오 5종

#시나리오처리
1기가입회원이 로그인된 상태로 진입 → 쿠폰 다운로드 → 구매 (제휴쿠폰 보유자)유입(월1) + 쿠폰(월1) + 구매(주문수) 적립
2비회원 진입(난수코드 수집) → 이탈 → 다음날 동일 난수코드로 재진입 후 가입난수코드가 가입 시 회원ID와 매핑, 가입 1회 적립
3기가입자가 로그아웃 상태로 진입 → 쿠폰 다운로드 시도 → 로그인로그인 시 난수코드-회원ID 매핑 갱신, 쿠폰 적립
4동일 회원이 한 달에 3번 우리카드 APP에서 진입 (항상 동일 난수코드)유입 이벤트 3회 발생하나 "월 1회" 룰로 1회만 적립
5구매 후 다음 달에 주문 취소취소 발생 월 정산에서 차감 (취소 룰 적용)
