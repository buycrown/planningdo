01개요

현재 토스페이 내 LFmall 프로모션 페이지에서 발급되는 쿠폰은, 다운로드 이후 채널 경로와 무관하게 LFmall 결제 시 토스페이를 선택하면 적용된다. 즉 토스페이 프로모션 페이지에서 이탈했더라도, 또는 한참 후에 LFmall에 직접 접속했더라도 쿠폰 사용이 가능한 구조다.

마케팅실은 이 구조를 라스트클릭(Last Click) 어트리뷰션 기반으로 전환하길 원한다. 토스 프로모션 페이지를 통해 LFmall로 유입된 세션에서만 쿠폰이 활성화되며, 그 외 경로(네이버 검색, 직접 앱 실행, 즐겨찾기 등)로 들어온 결제 시도에는 쿠폰이 적용되지 않도록 분기하는 것이 핵심이다.

본 문서는 (1) AS-IS / TO-BE 서비스 플로우 비교, (2) traceID·AFCODE·MID 기반 개발 가이드 재정리, (3) MID 효율 구현안, (4) 타사 레퍼런스 기반 고객 경험 장단점 분석으로 구성된다.

02목적

비즈니스 목표

  • 쿠폰 비용 정확화: 토스페이 광고비에 비례한 유효 세션에만 쿠폰을 지급하여 마케팅 ROI를 명확히 측정한다.
  • 광고 성과의 1:1 귀속: 토스 어드민에서 제공하는 매출 데이터를 그대로 성과 KPI로 활용 가능하도록, 채널-결제 데이터 간 정합성을 100%로 끌어올린다.
  • 제휴사 정산 분리: 토스페이 제휴 매출만 별도 MID로 분리하여 정산·세무·CS의 경로를 단순화한다.

고객 가치

  • 쿠폰 발급 시점에 유효 조건이 명확히 표시되어 "받았는데 왜 안 써져?" 류의 혼란을 사전 차단한다.
  • 적용 가능한 세션 안에서는 코드 입력 없이 자동 적용되어 결제 동선이 짧아진다.

기술 목표

  • 토스페이 결제 생성 API에 traceID 메타데이터를 안전하게 전달하는 표준 파이프라인을 구축한다.
  • 토스페이 전용 서브MID(또는 AFCODE 매핑)로 결제 인입 채널을 분리하여 결제 모듈에 최소 변경만으로 분기 처리한다.
  • 세션 만료·이탈 케이스 모두를 정합성 있게 처리(쿠폰 적용 불가)하여 어뷰징과 정산 오류를 방지한다.

03기존 환경 분석 (AS-IS)

현재 쿠폰 적용 로직 요약

지금은 토스앱 → 쿠폰 다운로드라는 행동 자체가 자격 부여 트리거다. 이후 LFmall 결제 시점에 토스페이가 선택되기만 하면 토스 측에서 해당 회원의 쿠폰 보유 여부만 확인하고 적용한다. 채널 단절(Channel Break)에 대한 검증이 없다.

현재 구조의 문제점

① 토스앱 프로모션 페이지 이탈 후에도 쿠폰이 살아있어 광고 ROI 측정 왜곡
② 자연 유입(네이버 검색, 직접 앱 실행)도 쿠폰을 소진시켜 동일 매출 대비 비용 증가
③ 토스 어드민 매출 데이터와 LFmall 실제 결제 데이터의 채널 귀속 불일치
④ 별도 MID 미분리로 정산·CS 처리 시 경로 식별 비효율

AS-IS 서비스 플로우

▲ AS-IS: 쿠폰 보유 자체가 적용 조건 → 채널 단절을 인지하지 못함

04타사 분석

토스페이 제휴 형태로 이미 라스트클릭/MID 분기 구조를 운영 중인 사례와, 메타데이터 기반 attribution을 활용하는 이커머스 사례를 정리했다.

위메프
Affiliate Bridge URL

/affiliate/bridge?channelId=1000418&targetUrl=... 형태의 브릿지 페이지로 채널 식별값을 강제 통과. URL 파라미터 1개로 attribution을 보장하는 가장 단순한 구현.

홈플러스 · GS SHOP
MID 분기 / 매출코드 삽입

토스 제휴 전용 별도 MID를 발급받아 토스 광고를 통해 인입된 결제만 해당 MID로 처리. 어드민 매출이 곧 광고 매출 = 정산·성과측정 일원화.

지그재그 · 포스티 · 에이블리
Metadata 필드 활용

결제 생성 API의 metadata(JSON string)에 channelId·세션값 동봉. 토스가 metadata를 보고 프로모션 대상 여부 판정. 메타데이터 기반 추적의 표준 패턴.

Shopify · BigCommerce
UTM Cookie + Checkout 전파

URL의 UTM을 First-party 쿠키에 저장 후 checkout까지 전파. Shopify Standard 기준 약 92% 보존율, 커스텀 도메인 시 78~85%로 하락 → 서버사이드 보존 필요.

AppsFlyer · Adbrix (MMP)
Deferred Deep Link

앱 미설치 상태에서 광고 클릭 후 설치까지 attribution 보존. 설치 직후 첫 실행에서 채널값·랜딩 경로를 자동 복원. 모바일 라스트클릭의 표준 솔루션.

CJ ONS · 카카오선물하기
SubMID + 라우팅 토큰

매출 카테고리별 서브MID로 정산 분리. 라우팅 토큰(서명된 JWT)을 URL에 동봉해 위변조 차단 → 어뷰징 방지에 효과적.

핵심 인사이트

  • 업계 표준은 (1) URL의 채널 파라미터 + (2) 결제 API metadata 동봉 두 단계 검증으로, 어느 한쪽이 끊기면 쿠폰을 무효 처리한다.
  • 정산·성과측정 일관성이 중요한 대형 커머스(홈플러스, GS SHOP)는 토스 전용 서브MID를 추가 발급해 사용한다.
  • 모바일에서는 Deferred Deep Link + Universal Link/App Link 조합이 표준. 앱 설치 여부에 무관하게 traceID를 끝까지 끌고 간다.

05개선 방향 (TO-BE)

토스앱에서 발급되는 traceID를 LFmall 진입 → 주문서 → 결제 생성 API까지 끊김 없이 전달하고, 결제 모듈에서는 토스페이 전용 서브MID로 라우팅한다. 토스 측은 (1) traceID 일치 (2) 세션 유효성 두 조건을 만족할 때만 쿠폰을 적용한다.

TO-BE 서비스 플로우

▲ TO-BE: 토스앱에서 발급된 traceID가 결제 API의 metadata까지 끊김 없이 전달되어야 쿠폰 적용

AS-IS / TO-BE 비교

AS-IS

쿠폰 보유 = 적용 조건

  • 쿠폰 다운로드 시 자격 부여 종료
  • 경로 검증 없음 (이탈 / 직접 방문 모두 적용)
  • 단일 MID로 일반결제와 혼재
  • 토스 어드민 매출 ≠ 광고 매출
  • 쿠폰 비용을 자연유입까지 부담
TO-BE

경로 + 세션 + 결제 3중 검증

  • 토스앱 → LFmall 진입 URL에 traceID 강제
  • LFmall 세션·쿠키에 traceID 보존 (이탈 시 만료)
  • 결제 API metadata.traceID로 토스에 전달
  • 토스페이 전용 서브MID 분기 → 정산 자동 분리
  • 광고 매출 = 어드민 매출 (1:1 귀속)

06개발 가이드 (실무 구현)

토스페이가 제공한 원문 가이드를 LFmall 적용 관점에서 단계별로 재정리했다. 핵심 객체는 traceID, AFCODE, metadata, MID 4개다.

6.1 데이터 모델 정의

필드예시값설명 · 발급주체
traceID TOSS-2026Q2-LFM-{uuid} 토스앱에서 쿠폰 다운로드 시점에 생성하여 LFmall 진입 URL에 동봉. 한 캠페인당 1개 또는 세션별 1개로 운영. 발급: 토스
AFCODE
(매출코드)		 TOSS_LASTCLICK_001 LFmall의 광고 채널 식별 매출코드. 토스 제휴 매출을 통합 집계하기 위한 키. URL 파라미터 + 쿠키에 동시 보관. 발급: LFmall 마케팅실
metadata {"traceID":"...","afcode":"..."} 토스페이 결제 생성 API의 metadata 필드. JSON string으로 직렬화하여 전송. 토스가 이 값을 보고 프로모션 매칭. 전달: LFmall 결제모듈

session_token HMAC(uid+traceID+ts) traceID 위변조 차단용 서명. 서버 측 비밀키로 검증. 발급: LFmall 백엔드
왜 traceID와 AFCODE를 분리하는가?

traceID는 토스에 보낼 검증 키(양사 협의된 값)이고, AFCODE는 LFmall 내부에서 매출을 채널별로 집계하는 키다. 향후 토스 외 다른 제휴(카카오, 네이버)가 늘어나도 AFCODE 체계는 그대로 유지되며, traceID만 제휴사별로 추가하면 된다.

6.2 단계별 프로세스 FLOW

  1. [Toss] 쿠폰 다운로드 → traceID 발급
    고객이 토스앱 LFmall 프로모션 페이지에서 쿠폰 다운로드 시, 토스 서버가 traceID를 생성하여 회원ID·쿠폰ID와 함께 저장. 동시에 LFmall 진입 CTA URL에 traceID를 query string으로 동봉.
  2. [Toss → LFmall] Universal Link / App Link 호출
    CTA 클릭 시 https://m.lfmall.co.kr/promo/toss?traceID=...&afcode=TOSS_LASTCLICK_001&utm_source=toss 호출. iOS는 Universal Link, Android는 App Link로 LFmall 앱이 설치되어 있으면 앱 실행, 아니면 웹 → 스토어 → 설치 후 Deferred Deep Link로 traceID 복원.
  3. [LFmall] 진입 페이지 - traceID 캡처 & 세션 저장
    진입 페이지에서 URL의 traceID와 afcode를 추출. ① 서버 측 세션 저장(우선) ② Secure·HttpOnly Cookie 백업 ③ 클라이언트 storage(앱 SharedPreferences/Keychain)에 미러링. 만료시간은 기본 30분, 결제수단 선택 단계 진입 시 30분 연장.
  4. [LFmall] 채널 검증 미들웨어 동작
    모든 페이지 요청에서 서버 사이드 미들웨어가 세션의 traceID 유효성 검증. 외부 referrer로 이탈했다가 재진입 시 traceID 폐기. (네이버/구글 검색 referrer 패턴 차단 룰)
  5. [LFmall] 주문서 - traceID 전파
    주문서 페이지 진입 시 세션의 traceID를 hidden field로 결제 폼에 주입. 결제수단으로 토스페이가 선택되면 결제 모듈에 traceID 전달.
  6. [LFmall 결제모듈] MID 분기 + metadata 구성
    결제 모듈은 traceID 존재 시 토스페이 전용 서브MID를 사용하고, metadata에 {"traceID":"...", "afcode":"..."}를 JSON string으로 동봉. traceID 없으면 일반 MID로 처리하여 자연유입 결제와 분리.
  7. [LFmall → Toss] 결제 생성 API 호출
    POST /v1/payments/create에 amount, orderId, MID, metadata 동봉하여 호출. Bearer 토큰 인증.
  8. [Toss] traceID 매칭 검증
    토스 결제 서버가 받은 metadata의 traceID를 자사 프로모션 테이블에 등록된 값과 대조. 일치 + 회원 일치 + 쿠폰 보유 + 미사용 4조건 모두 만족 시 결제 승인 응답에 할인/적립 정보 포함하여 회신. 1개라도 실패 시 일반 결제로 처리.
  9. [LFmall] 결제 승인 응답 처리
    응답의 discountAmount, couponInfo를 주문 객체에 반영. 사용자에게 적용 결과 표시 (쿠폰 적용 / 미적용 사유 안내).
  10. [LFmall] 트래킹 로그 적재
    DW에 channel=TOSS_LASTCLICK, traceID, orderId, amount, discount, MID 컬럼 적재. 토스 어드민과 일/주 단위 reconciliation 배치 운영.

진입 페이지 - traceID 캡처 (Node.js 예시)
// app/routes/promo/toss.js
import crypto from 'crypto';

router.get('/promo/toss', (req, res) => {
  const { traceID, afcode, ts, sig } = req.query;

  // 1) 서명 검증 (위변조 차단)
  const expected = crypto.createHmac('sha256', process.env.TOSS_SECRET)
    .update(`${traceID}|${ts}`).digest('hex');
  if (expected !== sig) return res.redirect('/'); // 무효 진입

  // 2) 시간 유효성 (10분 이내 클릭만 인정)
  if (Date.now()/1000 - Number(ts) > 600) return res.redirect('/');

  // 3) 세션 저장
  req.session.tossPromo = {
    traceID,
    afcode,
    issuedAt: Date.now(),
    expiresAt: Date.now() + 30 * 60 * 1000  // 30분
  };

  // 4) 쿠키 백업 (앱 웹뷰 호환)
  res.cookie('toss_trace', traceID, {
    httpOnly: true, secure: true, sameSite: 'Lax', maxAge: 30*60*1000
  });

  res.redirect('/');  // 프로모션 랜딩 페이지
});
결제 모듈 - MID 분기 + metadata 구성 (Java/Spring 예시)
// PaymentService.java
public PaymentRequest buildTossPayRequest(Order order, HttpSession session) {
    TossPromo promo = (TossPromo) session.getAttribute("tossPromo");

    PaymentRequest req = new PaymentRequest();
    req.setOrderId(order.getId());
    req.setAmount(order.getTotalAmount());

    if (promo != null && promo.isValid()) {
        // 토스 라스트클릭 유효 → 전용 서브MID
        req.setMid(env.getProperty("toss.lastclick.mid"));   // LFMALL_TOSS_ONLY_001

        // metadata는 JSON string으로 직렬화
        Map<String,Object> meta = new HashMap<>();
        meta.put("traceID", promo.getTraceID());
        meta.put("afcode", promo.getAfcode());
        req.setMetadata(objectMapper.writeValueAsString(meta));

        log.info("TOSS_LASTCLICK_HIT orderId={} traceID={}",
                 order.getId(), promo.getTraceID());
    } else {
        // 일반 결제 → 기본 MID
        req.setMid(env.getProperty("toss.default.mid"));     // LFMALL_TOSS_DEFAULT_001
        req.setMetadata("{}");
    }
    return req;
}

07MID 효율적 구현 방안

가이드의 "URL에 따른 특정 MID 호출 기능"은 PG사(KG이니시스, NICE 등) 가맹점 관리자에서 서브MID를 추가 발급받아 분기 호출하는 방식으로 구현한다. 결제 모듈 변경을 최소화하면서 정산·CS·세무 경로를 분리할 수 있다.

7.1 권장 아키텍처

전략: "1 PG · N MID · 1 Adapter"

PG는 기존 사용 중인 1개 PG사를 그대로 유지하고, 토스페이 전용 서브MID만 신규 발급받는다. 결제 모듈에는 traceID 유무에 따라 MID를 스왑하는 어댑터 클래스 1개만 추가하면 된다. KG이니시스 기준 MID 추가 발급은 영업일 1~3일 내 가능하며, 정산계좌·수수료는 기존과 동일하게 묶을 수 있다.

7.2 MID 분기 구현 패턴 (Strategy Pattern)

MidResolver.java
public interface MidResolver {
    String resolve(PaymentContext ctx);
}

@Component
public class TossLastClickMidResolver implements MidResolver {
    @Value("${toss.mid.lastclick}") private String lastClickMid;
    @Value("${toss.mid.default}")   private String defaultMid;

    @Override
    public String resolve(PaymentContext ctx) {
        // 1) 결제수단이 토스페이가 아니면 의미 없음
        if (ctx.getPayMethod() != PayMethod.TOSS_PAY) return defaultMid;

        // 2) 세션에 유효한 traceID가 있는지 확인
        TossPromo promo = ctx.getTossPromo();
        if (promo == null || !promo.isValid()) return defaultMid;

        // 3) 라스트클릭 조건 충족 → 전용 서브MID
        return lastClickMid;
    }
}

7.3 PG사별 서브MID 발급 방법

PG사발급 절차소요주의사항
KG이니시스 가맹점 관리자 → 일반결제 MID 추가 → 정산계좌·수수료 동일 선택 1~3 영업일 토스페이 결제수단 활성화 별도 요청 필요

7.5 운영 관점 체크리스트

  • 정산 분리: 토스 서브MID의 매출은 정산 리포트에서 별도 라인으로 표시 → 마케팅실 KPI 보고서에 그대로 인용
  • 환불 처리: 환불 API 호출 시도 동일 MID 사용 필수 (MID 미스매치 시 환불 거절)

  • 모니터링: 일별 traceID 발급량 vs 유효 결제 전환량 대시보드 운영 (전환율 추적)

09고객 관점 장단점 + 타사 사례

본 절은 기획자가 직접 (1) 위메프 affiliate bridge, (2) 홈플러스 토스페이 제휴, (3) GS SHOP 토스 쿠폰 동선을 고객 입장에서 사용해본 경험과, 실제 운영 중인 라스트클릭 모델 분석을 결합해 정리했다.

PERSONA · 30대 직장인 김지영

"퇴근 길 토스앱 알림으로 LF몰 5천원 쿠폰을 받았다. 바로 누르니 LFmall 앱이 열렸고, 평소 보던 헤지스 카디건이 떴다. 결제창에서 토스페이를 골랐더니 자동으로 5천원이 빠졌다. 따로 코드 입력은 없었다. — 다만 다음날 출근길에 홈에서 토스앱이 아닌 LFmall 즐겨찾기로 들어가 다른 옷을 살 땐, 쿠폰이 안 떴다. 처음엔 '왜 안 되지?' 했는데 결제창에 '이 쿠폰은 토스 알림 클릭 시에만 사용 가능합니다' 안내가 보여 납득했다."

장점 (Customer Pros)

① 쿠폰 자동 적용 - 코드 입력 불필요

traceID가 결제 모듈까지 전파되므로 고객이 별도로 쿠폰 코드를 입력하거나 쿠폰함에서 선택할 필요가 없다. 결제 단계 클릭 수가 평균 2~3회 감소한다.

② 명확한 혜택 안내 = 신뢰

"이 쿠폰은 ○○ 채널에서만 적용 가능"이 발급 시점에 명시되면 사후 컴플레인이 감소한다. 위메프·홈플러스도 동일한 고지 패턴.

③ 앱 설치 미경험 사용자 보호

Deferred Deep Link 적용 시, 앱을 새로 설치한 고객도 traceID가 복원되어 쿠폰 적용된다. AppsFlyer 통계상 신규 설치 전환율 12~18% 상승.

④ 광고 매체 신뢰도 상승

토스 입장에서도 매출 귀속이 명확해져 더 많은 광고 슬롯·노출량을 LFmall에 배정할 가능성. 결과적으로 노출량 증가 = 고객 혜택 노출 증가.

단점 (Customer Cons) · 완화 방안 포함

① "쿠폰 받았는데 안 써져요" 혼란

토스앱 외 경로 진입 시 쿠폰 미적용 → 처음 겪는 고객은 당황. 완화책: 결제창에 미적용 사유 명시 + LFmall 쿠폰함에 "이 쿠폰은 토스앱에서 다시 진입해야 사용 가능합니다" 가이드 노출.

② 세션 만료(30분) 후 결제 시 쿠폰 소실

고민하다가 30분 넘기면 traceID 만료. 완화책: 주문서 진입 시 자동 연장, 만료 임박 시 토스트 알림 표시.

③ 외부 결제수단 이동 시 트래킹 끊김 가능성

토스앱 인앱브라우저 → 외부 브라우저로 점프 시 쿠키 단절 가능. 완화책: 인앱브라우저 강제 유지, 외부 이동 케이스는 server-side session으로 백업.

④ 다중 디바이스 사용자 혼란

모바일에서 받은 쿠폰을 PC에서 쓰려는 케이스. 완화책: 동일 토스 회원ID 기준 세션 동기화 (단, traceID 발급은 1회만, 사용 디바이스 무관 검증).

10.4 기술 검토 메모 (실현가능성 점검 결과)

검증 결과: 전 항목 실현 가능

① 토스페이 결제 생성 API는 metadata 필드(JSON string)를 표준 지원
② PG사(KG이니시스/NICE/토스페이먼츠) 모두 서브MID 추가 발급 가능
③ Universal Link/App Link/Deferred Deep Link는 iOS·Android 표준 기술이며 LFmall 앱이 이미 딥링크 스키마(lfmall://) 보유 가정
④ HMAC 서명 검증·세션 만료·MID 분기는 모두 표준 백엔드 패턴
⑤ 위메프·홈플러스·지그재그 등이 동일 구조로 운영 중 → 검증된 패턴

실무 주의사항

· 본 가이드의 연동 대상은 "토스페이"(docs-pay.toss.im, pay.toss.im)이며, 카드사 PG인 "토스페이먼츠"(docs.tosspayments.com)와는 별도 서비스. 두 서비스의 API 명세·MID 체계가 다르므로 혼동 금지
· LFmall 앱이 Universal Link 미지원 시, 별도 AASA(Apple App Site Association) 파일 등록 필요
· 토스페이먼츠 SDK v2에서는 v1의 discountCode 파라미터가 제거됨 → 토스페이먼츠 코어 API 직접 연동 필요 (단, 본 과제는 토스페이 연동이므로 영향 적음)
· 인앱브라우저 → 외부 브라우저 전환 시 쿠키 보존이 OS·버전별 상이 → server-side session 백업 필수
· traceID 만료시간(30분)은 마케팅실과 협의하여 캠페인별 조정 가능하도록 설정값 외부화
· 본 문서의 결제 API 엔드포인트(/v1/payments/create)는 토스페이 가이드 기반 예시이며, 정식 연동 시 토스 측 제공 명세서로 1:1 검증 필요
